# 提示词编写指南 - 生成多样性样例

## 核心原则

### 1. 明确多样性要求

在提示词中**明确要求多样性**，而不是仅仅说"生成不同的问题"。具体说明：

```markdown
### 多样性要求（重要！）
为了确保生成的问题具有多样性，请遵循以下策略：
1. 词汇替换：使用同义词、近义词
2. 句式变换：改变语序和句式结构
3. 表达风格：尝试不同的表达风格
4. 避免重复：不要只是简单替换1-2个词
```

### 2. 提供具体的多样性策略

不要只说"生成不同的问题"，而要**提供具体的策略指导**：

#### 策略1：词汇替换
- "查询" → "找出"、"获取"、"列出"、"显示"、"检索"
- "用户" → "客户"、"会员"、"账户"、"使用者"

#### 策略2：句式变换
- 陈述句 ↔ 疑问句
- 主动语态 ↔ 被动语态
- "查询X的Y" → "Y属于X的有哪些"

#### 策略3：表达风格
- 口语化："给我看看"、"找一下"
- 正式："请查询"、"请统计"
- 技术性："检索"、"筛选"、"聚合统计"

### 3. 使用示例引导

提供**具体的示例**，让模型理解什么是"多样性"：

```markdown
### 示例对比

原问题："查询所有用户的姓名和邮箱"

好的改写（多样性高）：
- "列出所有用户的姓名和邮箱地址"
- "显示用户表中的姓名和邮箱字段"
- "获取所有用户的姓名和邮箱信息"

不好的改写（多样性低）：
- "查询所有用户的姓名和邮箱"（完全相同）
- "查询所有用户的姓名和email"（只改了一个词）
```

### 4. 明确禁止事项

明确说明**不要做什么**：

```markdown
### 避免以下情况
1. 不要只是简单替换1-2个词
2. 不要使用完全相同的句式结构
3. 不要生成语义不同的问题（如果是paraphrase类型）
4. 不要生成过于相似的问题
```

### 5. 使用温度参数

在代码中调整temperature参数：
- **高温度（0.7-0.9）**：用于生成多样化的变体
- **低温度（0.1-0.3）**：用于生成准确的SQL

## 变体类型详解

### 1. paraphrase（改写）

**目标**：语义完全相同，表达完全不同

**提示词要点**：
- 强调"语义完全相同"
- 强调"表达方式完全不同"
- 提供多种改写策略
- 明确禁止简单替换

**示例提示词**：
```markdown
请生成一个语义完全相同但表达方式完全不同的问题。

多样性要求：
1. 词汇替换：使用同义词、近义词替换关键词
2. 句式变换：改变语序和句式结构
3. 表达风格：可以尝试不同的表达风格
4. 避免重复：不要只是简单替换1-2个词，要彻底改变表达方式
```

### 2. similar（相似）

**目标**：相似但略有不同，查询相关数据

**提示词要点**：
- 说明可以改变什么（条件、范围、维度）
- 说明不能改变什么（核心查询意图）
- 提供具体的改变方向

**示例提示词**：
```markdown
请生成一个相似但略有不同的问题。

可以通过以下方式创造多样性：
1. 条件变化：添加或修改查询条件
2. 查询范围变化：扩大或缩小查询范围
3. 查询维度变化：从不同角度查询相同数据
4. 复杂度调整：稍微增加或减少查询复杂度
```

### 3. different_style（不同风格）

**目标**：表达风格完全不同

**提示词要点**：
- 明确列出不同的风格类型
- 要求随机选择风格
- 提供每种风格的具体特征

**示例提示词**：
```markdown
请选择以下一种风格（随机选择，确保多样性）：

1. 口语化风格：使用日常口语表达
2. 正式商务风格：使用正式、专业的表达
3. 技术性风格：使用技术术语和精确表达
4. 简洁风格：使用最简洁的表达
5. 详细描述风格：使用更详细的描述
```

### 4. different_angle（不同角度）

**目标**：从不同维度查询相同或相关数据

**提示词要点**：
- 列出不同的查询维度
- 说明可以从哪些角度变化
- 提供具体的角度示例

**示例提示词**：
```markdown
可以从以下角度思考：

1. 时间维度变化：从不同时间范围查询
2. 统计维度变化：从总数 → 平均值 → 最大值
3. 筛选维度变化：从不同字段筛选
4. 排序维度变化：按不同字段排序
5. 业务视角变化：从不同业务角色视角
```

## 高级技巧

### 1. 使用多样性提示（diversity_hint）

在代码中动态添加提示，引导模型生成不同方向的变体：

```python
if variation_type == 'different_style':
    styles = ['口语化', '正式商务', '技术性', '简洁', '详细描述']
    diversity_hint = f"\n提示：请使用'{styles[j % len(styles)]}'风格"
```

### 2. 组合多种策略

不要只使用一种变体类型，而是**组合使用多种类型**：

```python
variation_types = [
    'paraphrase',      # 语义相同但表达不同
    'similar',         # 相似但略有不同
    'different_style', # 不同表达风格
    'different_angle', # 不同查询角度
]
```

### 3. 迭代优化

根据生成结果调整提示词：
- 如果生成的问题太相似 → 加强多样性要求
- 如果生成的问题语义偏差太大 → 强调语义一致性
- 如果生成的问题不自然 → 强调自然流畅

### 4. 使用Few-shot示例

在提示词中提供**好的和不好的示例对比**：

```markdown
### 示例对比

✅ 好的改写（多样性高）：
- "列出所有用户的姓名和邮箱地址"
- "显示用户表中的姓名和邮箱字段"

❌ 不好的改写（多样性低）：
- "查询所有用户的姓名和邮箱"（完全相同）
- "查询所有用户的姓名和email"（只改了一个词）
```

## 完整示例

### 改写类型（paraphrase）的完整提示词

```markdown
# 问题改写任务 - 语义相同但表达不同

## 数据库表结构
[schema_text]

## 原始样例
问题: [original_question]
SQL: [original_sql]

## 任务要求
请生成一个**语义完全相同**但**表达方式完全不同**的问题。

### 多样性要求（重要！）
为了确保生成的问题具有多样性，请遵循以下策略：

1. **词汇替换**：
   - 使用同义词、近义词替换关键词
   - 例如："查询" → "找出"、"获取"、"列出"、"显示"
   - 例如："用户" → "客户"、"会员"、"账户"

2. **句式变换**：
   - 改变语序和句式结构
   - 例如：陈述句 ↔ 疑问句
   - 例如：主动语态 ↔ 被动语态
   - 例如："查询X的Y" → "Y属于X的有哪些"

3. **表达风格**：
   - 可以尝试不同的表达风格（正式/口语/技术性）
   - 但保持语义不变

4. **避免重复**：
   - 不要只是简单替换1-2个词
   - 要彻底改变表达方式，让人感觉是完全不同的问题

### 示例对比

✅ 好的改写：
- 原问题："查询所有用户的姓名和邮箱"
- 改写1："列出所有用户的姓名和邮箱地址"
- 改写2："显示用户表中的姓名和邮箱字段"
- 改写3："获取所有用户的姓名和邮箱信息"

❌ 不好的改写：
- "查询所有用户的姓名和邮箱"（完全相同）
- "查询所有用户的姓名和email"（只改了一个词）

### 输出要求
- 只返回改写后的问题，不要返回SQL
- 问题应该自然流畅，符合中文表达习惯
- 确保语义与原问题完全一致

改写后的问题：
```

## 总结

编写好的提示词生成多样性样例的关键：

1. ✅ **明确要求多样性** - 不要只是说"生成不同的问题"
2. ✅ **提供具体策略** - 给出词汇替换、句式变换等具体方法
3. ✅ **使用示例引导** - 展示好的和不好的示例
4. ✅ **明确禁止事项** - 说明不要做什么
5. ✅ **调整温度参数** - 使用较高的temperature增加随机性
6. ✅ **组合多种类型** - 使用多种变体类型组合
7. ✅ **动态添加提示** - 使用diversity_hint引导生成方向
