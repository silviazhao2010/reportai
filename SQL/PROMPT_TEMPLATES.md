## 模板2：样例扩展提示词（Paraphrase类型）

### 使用场景
生成语义相同但表达方式完全不同的问题变体。

### 完整模板

```markdown
# 问题改写任务 - 语义相同但表达不同

## 数据库表结构

[此处插入数据库schema]

## 数据库业务规则和约定

[此处插入业务规则，如果有]

## 原始样例
问题: [原始问题]
SQL: [原始SQL]

## 任务要求
请生成一个**语义完全相同**但**表达方式完全不同**的问题。

### 多样性要求（重要！）
为了确保生成的问题具有多样性，请遵循以下策略：

1. **词汇替换**：
   - 使用同义词、近义词替换关键词
   - 例如："查询" → "找出"、"获取"、"列出"、"显示"、"检索"
   - 例如："用户" → "客户"、"会员"、"账户"、"使用者"
   - 例如："统计" → "计算"、"汇总"、"合计"、"计数"

2. **句式变换**：
   - 改变语序和句式结构
   - 例如：陈述句 ↔ 疑问句
   - 例如：主动语态 ↔ 被动语态
   - 例如："查询X的Y" → "Y属于X的有哪些"
   - 例如："统计X的数量" → "X有多少个"

3. **表达风格**：
   - 可以尝试不同的表达风格（正式/口语/技术性）
   - 但保持语义不变
   - 正式风格："请查询"、"请统计"、"请列出"
   - 口语风格："给我看看"、"找一下"、"有哪些"
   - 技术风格："检索"、"筛选"、"聚合统计"

4. **避免重复**：
   - 不要只是简单替换1-2个词
   - 要彻底改变表达方式，让人感觉是完全不同的问题
   - 确保改写后的问题与原问题在表达上差异明显

### 示例对比

✅ 好的改写（多样性高）：
- 原问题："查询所有用户的姓名和邮箱"
- 改写1："列出所有用户的姓名和邮箱地址"
- 改写2："显示用户表中的姓名和邮箱字段"
- 改写3："获取所有用户的姓名和邮箱信息"
- 改写4："用户表中包含哪些姓名和邮箱数据？"

❌ 不好的改写（多样性低）：
- "查询所有用户的姓名和邮箱"（完全相同）
- "查询所有用户的姓名和email"（只改了一个词）
- "查询所有用户的name和email"（只是中英文替换）

### 输出要求
- 只返回改写后的问题，不要返回SQL
- 问题应该自然流畅，符合中文表达习惯
- 确保语义与原问题完全一致
- 表达方式要与原问题明显不同

改写后的问题：
```

```markdown
### 多样性要求（重要！）
可以通过以下方式创造多样性：

1. **条件变化**：
   - 添加或修改查询条件
   - 改变数值范围（如：最近7天 → 最近30天）
   - 改变筛选条件（如：状态、类型等）
   - 例如："查询活跃用户" → "查询非活跃用户"
   - 例如："查询最近7天的订单" → "查询最近30天的订单"

   - 例如："本月" → "本季度"、"今年"

   - 从不同角度查询相同数据
   - 例如：按时间查询 → 按地区查询
   - 例如：统计总数 → 统计平均值
   - 例如：按部门分组 → 按职位分组

4. **复杂度调整**：
   - 可以稍微增加或减少查询复杂度
   - 但不要完全改变查询意图
   - 例如：添加排序 → 移除排序
   - 例如：添加分组 → 移除分组

- "查询最近7天注册的用户"（完全相同）
```
